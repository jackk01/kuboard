# Kuboard 实施路线图

## 1. 交付策略

建议采用“先打控制平面底座，再逐步加深体验”的路线，而不是一次性追求功能大而全。

总体节奏建议：

- Phase 0：技术底座与设计固化
- Phase 1：MVP
- Phase 2：增强版
- Phase 3：企业级扩展

## 2. Phase 0：基础设计与工程底座

目标：

- 把架构边界、规范、工程模板、部署方式定下来

交付物：

- Vue 前端工程骨架
- Django 后端工程骨架
- SQLite / Redis / Celery / ASGI 基础设施接入
- 登录认证最小闭环
- kubeconfig 解析与安全校验模块
- SQLite WAL、备份、恢复与完整性校验方案
- 基础 CI/CD、Lint、测试框架
- Helm 部署骨架

完成标准：

- 本地与测试环境可一键启动
- 基础登录、系统健康检查、配置管理跑通

## 3. Phase 1：MVP

目标：

- 让 Kuboard 可以真正纳管多集群并完成通用资源查看与编辑

范围：

- kubeconfig 导入集群
- 集群概览与健康状态
- API Discovery 与 OpenAPI 缓存
- 通用资源浏览器
- 资源详情页
- YAML 编辑器
- Dry-run + Server-Side Apply
- 基于 Kubernetes RBAC 的权限探测
- 审计日志
- RBAC、NetworkPolicy、Quota 类高价值资源优先体验

完成标准：

- 可以管理至少 2 个以上集群
- 可以查看内置资源与常见 CRD
- 可以对 Deployment、Service、ConfigMap、Ingress、Secret 等常见资源做查看和编辑
- 权限控制在前后端一致生效

## 4. Phase 2：增强版

目标：

- 强化实时性、排障效率和多用户协作体验

范围：

- Watch 实时刷新
- Events 聚合
- Pod Logs
- Pod Exec / Terminal
- 常用工作负载快捷页
- 收藏、最近访问、保存筛选视图
- Prometheus 指标接入
- 更丰富的审计筛选与导出

完成标准：

- 排障场景无需频繁切换 kubectl
- 大部分日常巡检和资源调整可在 Web 中完成

## 5. Phase 3：企业级扩展

目标：

- 进入平台化演进，沉淀治理和扩展能力

范围：

- 插件系统
- 受控 kubeconfig 导出
- GitOps 联动
- 策略中心
- 告警中心
- 多集群应用分发
- 多租户细粒度配额与隔离增强

完成标准：

- Kuboard 不只是控制台，而是可扩展的多集群运维平台

## 6. 推荐研发分工

建议最小团队配置：

- 1 名产品/项目负责人
- 1 名 UI/UX 设计师
- 2 名前端工程师
- 2 名后端工程师
- 1 名测试或质量工程师
- 1 名 DevOps / 平台工程师

如果团队更小，优先保证：

- 1 名前端
- 2 名后端
- 1 名兼测试/发布的工程师

## 7. 技术风险与应对

### 7.1 kubeconfig 安全风险

风险：

- 恶意 kubeconfig 可能触发外部命令执行或敏感文件暴露

应对：

- 默认禁用 `exec` 插件
- 强制安全审计和白名单
- 凭证加密存储

### 7.2 大集群性能风险

风险：

- 列表和 Watch 可能压垮 API Server 或 Kuboard 后端

应对：

- 服务端 Watch 汇聚
- 分页、缓存、限流
- 压测和容量规划

### 7.3 权限一致性风险

风险：

- 前端显示与后端真正权限不一致，导致体验混乱

应对：

- 一切以 Kubernetes 权限探测结果为准
- 页面级能力和接口级权限共用同一判定逻辑

### 7.4 CRD 兼容性风险

风险：

- 不同 CRD Schema 质量参差不齐，自动表单渲染可能不稳定

应对：

- YAML 编辑保持为一等能力
- 表单编辑优先覆盖内置资源和高频 CRD
- 插件系统为复杂 CRD 提供专用渲染器

### 7.5 SQLite 并发与恢复风险

风险：

- SQLite 在多副本共享写、高频审计写入和异常断电场景下更脆弱

应对：

- 后端单实例部署
- 启用 WAL、定时备份和恢复演练
- 把会话、缓存、队列和审计明细迁到 Redis 或外部归档

## 8. 里程碑建议

### M1

- 完成工程骨架、登录和集群导入
- 完成 SQLite 备份恢复闭环

### M2

- 完成 Discovery、OpenAPI、资源浏览器、详情页

### M3

- 完成 YAML 编辑、Dry-run、Apply、审计

### M4

- 完成 Watch、Logs、Exec 和性能优化

### M5

- 完成插件化扩展与企业能力

## 9. 验收清单

- 多集群导入是否稳定
- 内置资源与 CRD 是否可浏览
- YAML 编辑是否可正确校验与提交
- 权限不足是否准确提示
- 审计日志是否可追踪关键动作
- 大集群场景下是否仍可接受
- 平台故障是否具备定位手段
- SQLite 恢复演练是否通过

## 10. 下一步建议

文档阶段结束后，建议立刻进入以下工作：

1. 初始化前后端工程
2. 定义 API 契约与 OpenAPI 文档
3. 落地 kubeconfig 导入与安全校验
4. 搭建通用资源浏览器 MVP
5. 接入 RBAC 权限探测与审计链路
